# Валидация качества ответов в Валли

**Дата реализации:** 15 декабря 2024  
**Статус:** ✅ Реализовано (требуется тестирование и deploy)  
**Автор:** AI Code Agent

## Описание

Добавлена **двухпроходная валидация** ответов пользователя на вопросы Валли. Система теперь проверяет каждый ответ на соответствие минимальным критериям перед переходом к следующему вопросу.

## Проблема

**До:**
- Валли задавал 7 вопросов по фиксированному счётчику
- Если пользователь писал нерелевантную чушь или слишком общие ответы — они засчитывались
- Шаг (`current_step`) увеличивался автоматически после каждого сообщения
- Качество финального скоринга страдало

**После:**
- Валли **добивается качественных ответов** перед переходом к следующему вопросу
- Шаг увеличивается **только** если ответ достаточен
- При недостаточном ответе — мягкое уточнение с примерами

## Архитектура решения

### Вариант C: Двухпроходная проверка (реализовано)

```
Пользователь → Сообщение
    ↓
Edge Function (mode=dialog)
    ↓
┌─────────────────────────────────────┐
│ 1. GENERATOR (Валли-ассистент)     │
│    Вход: messages, current_step    │
│    Выход: черновой ответ Валли     │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 2. VALIDATOR (строгий оценщик)      │
│    Вход: ответ пользователя,       │
│           критерии шага             │
│    Выход: is_sufficient + feedback  │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ РЕШЕНИЕ:                            │
│ • is_sufficient=true                │
│   → current_step++                  │
│   → вернуть ответ генератора        │
│                                     │
│ • is_sufficient=false               │
│   → current_step не меняется        │
│   → вернуть уточнение               │
└─────────────────────────────────────┘
```

### Fail-safe логика

- **При ошибке валидатора** → блокируем продвижение шага (безопаснее)
- **Первое сообщение** → валидатор не вызывается (Валли задаёт первый вопрос)
- **Шаг > 7** → считаем валидацию завершённой

## Минимальные критерии по шагам

### Шаг 1 — Суть идеи
**Вопрос:** "Расскажи о своей идее. Что ты создаёшь?"

**Критерии:**
- Что конкретно создаётся (продукт/услуга)
- Какой результат получит пользователь
- Краткое описание сути (1-2 предложения)

**Пример шаблона:**
```
Я создаю мобильное приложение для учёта личных финансов.
Пользователь сможет видеть все расходы в одном месте и получать рекомендации по экономии.
Это поможет ему не тратить лишнего и копить на цели.
```

---

### Шаг 2 — Проблема
**Вопрос:** "Какую проблему решает твоя идея? Кто от неё страдает?"

**Критерии:**
- Кто страдает (роль/тип человека/компании)
- Как часто / в каком сценарии возникает боль
- Почему это реально больно сейчас (последствия: деньги/время/стресс/риски)

**Пример шаблона:**
```
Фрилансеры и самозанятые теряют деньги, потому что не контролируют мелкие расходы.
Каждый день они тратят 2000-3000 тенге на кофе, обеды, такси — в конце месяца не хватает на аренду.
Это вызывает стресс и мешает копить на развитие бизнеса.
```

---

### Шаг 3 — Целевая аудитория
**Вопрос:** "Для кого конкретно твоя идея? Кто твоя целевая аудитория?"

**Критерии:**
- Кто пользователь, кто покупатель (если отличаются)
- Сегмент/роль (микро-сегмент, не "все предприниматели")
- Где найти этих людей (каналы: сообщества/площадки/поиск/офлайн)

**Пример шаблона:**
```
Фрилансеры из Алматы и Астаны, 25-35 лет, зарабатывают 300-700 тыс тенге в месяц.
Работают в дизайне, маркетинге, IT.
Ищу их в телеграм-каналах для фрилансеров, на Kwork, в коворкингах.
```

---

### Шаг 4 — Валидация знаний
**Вопрос:** "Как ты узнал об этой проблеме? Что уже проверял?"

**Критерии:**
- Откуда знание о проблеме (личный опыт / разговоры / наблюдения)
- Что уже пробовали проверить (если ничего — так и пишем)
- С кем разговаривал / какие сигналы получил

**Пример шаблона:**
```
Я сам фрилансер, постоянно сталкиваюсь с этой проблемой.
Поговорил с 5 коллегами из коворкинга — все жалуются на то же самое.
Запустил опрос в телеграм-канале — 80% ответили, что не контролируют расходы.
```

---

### Шаг 5 — Конкуренты и альтернативы
**Вопрос:** "Как люди решают эту проблему сейчас? Кого считаешь конкурентами?"

**Критерии:**
- Как решают сейчас (ручные обходные пути/замены)
- Почему текущие способы плохи (1-2 причины)
- Кого считаешь альтернативой/конкурентом (хотя бы 1 пример или "нет прямых, но есть замены")

**Пример шаблона:**
```
Сейчас люди записывают расходы в блокнот или Excel, но это неудобно и забывают вносить.
Есть приложения типа 1Money, но они сложные и перегружены функциями.
Некоторые вообще ничего не делают и просто тратят всё.
```

---

### Шаг 6 — Уникальное преимущество
**Вопрос:** "Почему именно ты? В чём твоё уникальное преимущество?"

**Критерии:**
- Почему вас сложно скопировать (скорость/доступ/компетенция/данные/комьюнити)
- Что делаете лучше/проще текущих альтернатив
- Личное преимущество (опыт/связи/знания)

**Пример шаблона:**
```
Я 5 лет работаю фрилансером, понимаю боли изнутри.
Уже есть аудитория в 2000 подписчиков в телеграме.
Сделаю самый простой интерфейс — 3 кнопки вместо 20 функций.
```

---

### Шаг 7 — Риски и план действий
**Вопрос:** "Что может убить твою идею? Какой главный риск?"

**Критерии:**
- Главный риск (спрос/канал/юнит-экономика/право/доступ)
- План валидации на 24-72 часа (1-2 быстрых эксперимента)
- Что сделаешь, если гипотеза не подтвердится (пивот/упрощение/другой сегмент)

**Пример шаблона:**
```
Главный риск — люди не захотят платить за ещё одно приложение.
Проверю это за 3 дня: запущу пост с офером "первым 50 — бесплатно навсегда".
Если не наберу 20 заявок — упрощу до телеграм-бота вместо приложения.
```

## Тестирование

### Ручное e2e-тестирование (обязательно)

**Сценарий 1: Нерелевантный ответ блокирует продвижение**

1. Начать новую валидацию с Валли
2. На вопрос "Расскажи о своей идее" написать: "Идея топ"
3. **Ожидаемое поведение:**
   - Валли возвращает уточнение: "Пока слишком общо..."
   - `current_step` остаётся = 1
   - В БД `idea_validations.current_step = 1`
4. Написать снова "Хорошая идея"
5. **Ожидаемое поведение:**
   - Снова уточнение
   - Шаг не увеличивается

**Сценарий 2: Релевантный ответ пропускает**

1. Продолжить ту же сессию
2. Написать развёрнутый ответ с деталями (см. примеры шаблонов)
3. **Ожидаемое поведение:**
   - Валли принимает ответ
   - Переходит к вопросу 2 (проблема)
   - `current_step` = 2 в БД

**Сценарий 3: Прохождение всех 7 вопросов**

1. Пройти все 7 вопросов с качественными ответами
2. **Ожидаемое поведение:**
   - Каждый шаг увеличивается только при достаточном ответе
   - После шага 7 → переход к скорингу
   - `current_step` = 7, затем `status = 'completed'`

**Сценарий 4: Fail-safe при ошибке валидатора**

1. (Требует искусственной эмуляции сбоя)
2. **Ожидаемое поведение:**
   - Шаг НЕ увеличивается
   - Возвращается стандартное уточнение
   - Лог: `ERR validator_failed`

### Проверка логов (Supabase Dashboard)

После ручного тестирования проверить логи Edge Function:

```
INFO step_advanced { validationId, from: 1, to: 2 }
INFO step_blocked { validationId, currentStep: 1, isValid: false }
ERR validator_failed { message, step }
```

### Метрики для мониторинга

- **Средний процент блокировок** (step_blocked / всего сообщений)
- **Среднее количество попыток** на шаг (сколько раз пользователь отвечает на один вопрос)
- **Процент ошибок валидатора** (ERR validator_failed)

Цель: < 10% ошибок валидатора, 20-40% блокировок (норма для качественной валидации).

## Риски и ограничения

### Риски

1. **Средняя вероятность:** Модель-валидатор может быть слишком строгой или слишком мягкой
   - **Митигация:** Мониторинг метрик, корректировка промпта при необходимости

2. **Низкая вероятность:** Удвоенное количество вызовов модели → увеличение стоимости/латентности
   - **Митигация:** Валидатор настроен на `temperature=0.3`, `max_tokens=600` (экономия)

3. **Низкая вероятность:** Fail-safe блокирует все шаги при системных проблемах
   - **Митигация:** Логирование всех ошибок, алерты в Sentry

### Ограничения

1. **Субъективность:** Критерии "достаточности" зависят от модели
2. **Латентность:** +1-2 секунды на каждое сообщение (второй вызов API)
3. **Нет retry-логики:** Пользователь должен сам переформулировать ответ

## Дальнейшие улучшения (опционально)

1. **A/B тест:** Сравнить конверсию с валидацией vs без валидации
2. **Адаптивные критерии:** Снижать строгость после 3+ попыток на один вопрос
3. **Кэширование валидации:** Не проверять повторно одинаковые ответы
4. **Fine-tuning валидатора:** Обучить отдельную модель на примерах хороших/плохих ответов
